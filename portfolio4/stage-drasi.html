Stage chez Rectorat DRASI – Développement automatisation

Du 19 mai au 27 juin 2026 – Tuteur : Grégory Quinet

Présentation de l’entreprise

DRASI est une entreprise spécialisée dans le développement de solutions digitales sur mesure, notamment dans l’écosystème Salesforce. Elle accompagne ses clients dans la transformation digitale de leurs processus métiers grâce à des applications performantes, modulaires et évolutives.

L’équipe allie expertise technique, méthodologie agile et écoute client pour livrer des solutions adaptées aux besoins spécifiques de chaque projet.

Objectif du stage

Stage au Rectorat de Lille – Direction Régionale des Systèmes d’Information (DRASI)

Mission principale : Participation au développement, à la maintenance et aux tests automatisés d’une application de gestion du temps partiel pour les enseignants.

Environnement technique : Docker, PHP, MySQL, Cypress, Terraform, Composer, Git.

3. Tests Automatisés avec Cypress

Approche Technique avec Cypress

Exemples concrets de tests automatisés et bonnes pratiques mises en œuvre pendant le stage.

1. Structure des Scénarios de Test

Les scénarios Cypress suivent une logique Given/When/Then pour une lisibilité optimale et une maintenance facilitée.

Feature: Test d'accès
En tant qu'enseignant
Je souhaite utiliser l'application

Background:
  Given Ouverture application en profil "enseignant"

Scenario: J'arrive sur l'index
  Then Je vois la page index

Scenario: Désigner à un bureau de gestion
  Then Nom du bureau est visible
  Then Mail est valide

2. Interaction avec les Éléments du DOM

Utilisation de sélecteurs CSS et de méthodes Cypress pour interagir avec les éléments de l’interface.

// Clic sur un menu spécifique
When("Je clique sur le menu {string}", (menu) => {
  cy.get('input[value="' + menu + '"]').click();
});

When('J\'ouvre le menu {string}', (menu) => {
  cy.get(`input[value="${menu}"]`).click();
  cy.wait(500); // Attente pour stabiliser l'interface
});

3. Gestion des Boutons Radio

Vérification et sélection des boutons radio via un mapping pour éviter la duplication de code.

// Mapping des libellés vers les IDs des boutons radio
const radioMapping = {
  "Demande de temps partiel": "demande_nature__demande_tp",
  "Réintégration à temps complet": "demande_nature__reint",
  "Annulation": "demande_nature__annulation",
  "1er degré": "etab_affect__1degre",
  "2nd degré": "etab_affect__2degre",
  "Oui": "participation_mouvement__oui",
  "Non": "participation_mouvement__non"
};

Then("Le bouton radio {string} est présent", (label) => {
  cy.get(`input[type="radio"]#${radioMapping[label]}`)
    .should("exist")
    .and("be.visible");
});

When("Je sélectionne le bouton radio {string}", (id) => {
  cy.get(`input[type="radio"]#${id}`).check({ force: true });
});

Then("Le bouton radio {string} est coché", (id) => {
  cy.get(`input[type="radio"]#${id}`).should("be.checked");
});

4. Vérification des Formulaires et Pages

Validation de la présence des formulaires et des éléments clés sur chaque page.

// Vérification de la présence d'un formulaire
Then("Je vois le formulaire {string}", (formulaire) => {
  const mapping = {
    "Identité et situation": "#p1_identite",
    "Gestion d'un enseignant": "#v-selectionEnseignant",
    "Traitement des décisions": "#v-traitementDecision"
  };
  cy.get(mapping[formulaire]).contains(formulaire);
});

// Vérification des champs d'un formulaire
Then("Le champ {string} est visible", (champId) => {
  cy.get(`#${champId}`).should("be.visible");
});

// Exemple pour la page IEN
Then('Je vois la page index IEN', () => {
  cy.get(".blocBase")
    .contains("Synthèse des demandes de temps partiel pour la circonscription: ");
});

5. Gestion des Scénarios Complexes

Exemple de scénario complet pour tester la modification des champs de situation.

Scenario: Je modifie les champs de la situation
  When J'ouvre le menu "Ma demande"
  Then Je vois le formulaire "Identité et situation"
  When Je sélectionne le bouton radio "demande_nature__reint"
  Then Le bouton radio "demande_nature__reint" est coché
  When Je sélectionne le bouton radio "etab_affect__2degre"
  Then Le bouton radio "etab_affect__2degre" est coché
  When Je clique sur le bouton "Suivant"
  // Then Le formulaire suivant est affiché

6. Bonnes Pratiques et Difficultés Rencontrées

  • Mapping centralisé : Utilisation de constantes pour éviter la duplication de sélecteurs.
  • Attentes explicites : cy.wait(500) pour stabiliser l’interface après une action.
  • Forçage des actions : { force: true } pour les éléments parfois masqués par le CSS.
  • Difficultés :
    • Gestion des dates au format 0000-00-00 (corrigé en 0000-01-01 pour PHP8).
    • Problème de visibilité des demandes pour le profil IEN (lié aux circonscriptions).
    • Bouton "Suivant" parfois non cliquable (contourné avec { force: true }).

Saisie d’un numéro de dossier puis modification de la demande du numéro en question.

121_GestionTraitement - Compte gestionnaire - onglet “Traitement des décisions”

Test des boutons radios et traitement des utilisateurs.

201_enseignant - Compte enseignant - page d’accueil

Test la page d’accueil de l’enseignant et vérification de la présence des menus, du bureau et du mail.

211_IdentiteSituation - Compte enseignant - onglet “Ma demande”

Vérification des données de la personne connectée, test des boutons radios. Le bouton "suivant" ne fonctionne pas.

221_Recapitulatif - Compte enseignant - onglet “Récapitulatif”

Vérification de la présence du récapitulatif dans l’onglet.

301_ien - Compte IEN - page d’accueil

Vérification que tout soit présent sur la page.

Bilan et Difficultés

Réussites

  • Automatisation complète des tests pour les profils gestionnaire, enseignant et IEN.
  • Correction de bugs majeurs (dates SQL, accès aux données).

Limites

  • Certains tests Cypress non résolus (bouton “suivant”, recherche par numéro de dossier).
  • Page IEN : demandes non visibles malgré leur existence en base.
  • Traitement des décisions non fonctionnel en production.
